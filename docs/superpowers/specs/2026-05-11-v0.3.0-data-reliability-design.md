# v0.3.0 Data Reliability and Research Workflow Design

## Goal

v0.3.0 turns Taiwan Equity Lens from a usable fundamental-analysis CLI into a more trustworthy research workflow. The release focuses on making data status explicit, making workflow failures actionable, and preparing valuation outputs to depend on clearly labeled assumptions.

## Product Positioning

The project remains a research and screening tool, not an investment-advice engine. Its value should come from repeatable data collection, transparent quality checks, comparable stock reports, and visible assumptions behind valuation ranges.

For a user opening the repository or running the workflow, v0.3.0 should answer:

1. Which data sources were used?
2. How fresh is each input?
3. Which fields are missing or inferred?
4. Which workflow steps succeeded, failed, or were skipped?
5. Which valuation assumptions produced the target-price range?

## Scope

v0.3.0 will add a data reliability layer and improve the generated workflow summary. The release will keep the existing command style, static HTML outputs, CSV examples, and no-database architecture.

Included:

- data source status records for fetch, parse, price, valuation, and workflow steps
- freshness labels for source data when a date is available
- explicit missing-field and fallback-source warnings
- workflow summary improvements with step-level status, failure reasons, and retry hints
- report and dashboard presentation for data reliability state
- valuation assumption labeling for EPS and target-price scenarios
- documentation updates for the new reliability model

## Non-Goals

- Do not build a hosted service.
- Do not add user accounts, authentication, or a database.
- Do not add investment advice, buy/sell/hold labels, or portfolio allocation guidance.
- Do not guarantee real-time market data.
- Do not replace existing data sources unless current source contracts force it.
- Do not introduce a frontend framework.

## User Experience

### CLI Workflow

The existing one-command workflow remains the main path:

```powershell
python -m taiwan_stock_analysis.cli workflow --watchlist examples/watchlist.csv --out-dir dist
```

The command should still produce per-stock reports, comparison outputs, dashboard HTML, valuation CSV when requested, and a workflow summary. In v0.3.0, the summary should be useful enough to diagnose a run without opening every report.

### Workflow Summary

`workflow_summary.json` should expose a stable high-level structure:

- run metadata: timestamp, command inputs, output directory
- stock counts: requested, succeeded, failed, skipped
- step statuses: fetch, parse, analysis, valuation, report, comparison, dashboard
- data reliability summary: healthy, warning, error counts
- per-stock failures with reason, stage, and suggested next action

Existing consumers should not break. If the schema changes, existing fields must be preserved and new fields added alongside them.

### HTML Dashboard

The dashboard should surface workflow reliability before detailed report links:

- overall run status
- data warnings count
- failed or skipped stocks
- links to reports and workflow summary
- clear retry hints when failures are recoverable

### Single-Stock Report

Each report should include a data reliability section that explains:

- source links and source names
- data dates where available
- missing financial fields
- fallback behavior, such as TPEx price fallback
- valuation assumptions used for EPS and target-price scenarios

The report should remain static HTML and should continue embedding the full JSON payload.

## Architecture

Keep the project modular and small. Add reliability structures without forcing a large rewrite.

Recommended units:

- `src/taiwan_stock_analysis/reliability.py`
  - defines data status models, severity labels, freshness helpers, and retry hint mapping
- `src/taiwan_stock_analysis/workflow.py`
  - records step-level workflow status and per-stock failure context
- `src/taiwan_stock_analysis/report.py`
  - renders per-stock reliability details
- `src/taiwan_stock_analysis/dashboard.py`
  - renders workflow-level reliability summary
- `src/taiwan_stock_analysis/market_price.py`
  - reports source fallback and price date details through structured status records
- `src/taiwan_stock_analysis/valuation.py`
  - labels EPS and target-price assumptions in a structured way

The reliability layer should be plain dataclasses or dictionaries compatible with the current JSON outputs. It should avoid new dependencies.

## Data Model

Use a small status object that can be serialized into existing JSON outputs:

```json
{
  "stage": "price",
  "status": "warning",
  "source": "TPEx",
  "date": "2026-05-08",
  "message": "TWSE price was unavailable; TPEx fallback was used.",
  "retry_hint": "Run again after the next market data update."
}
```

Allowed `status` values:

- `ok`
- `warning`
- `error`
- `skipped`

Allowed severity interpretation:

- `ok`: data is present and no fallback was needed
- `warning`: usable output with missing fields, stale dates, or fallback source
- `error`: output for the stage could not be produced
- `skipped`: stage did not run because a prerequisite failed or the workflow did not require it

## Error Handling

Failures should be recorded in workflow outputs instead of only appearing in terminal text. A failed stock should not stop the whole watchlist unless the failure prevents all outputs.

Every recorded error should include:

- stock id when applicable
- failed stage
- short reason
- whether downstream steps were skipped
- suggested next action

## Testing

Tests should cover behavior rather than only snapshots:

- reliability status objects serialize to JSON-compatible dictionaries
- fallback price source produces a warning status
- stale or missing data date produces a warning status when detectable
- workflow summary preserves existing fields and adds step-level statuses
- failed stock rows include stage, reason, and retry hint
- dashboard renders warning and error counts
- report renders source status, missing fields, fallback source, and valuation assumptions
- all existing CLI tests continue passing

## Documentation

Update documentation for:

- README quick-start output explanation
- `docs/usage-workflow.md` workflow summary fields
- `docs/data-sources.md` source freshness and fallback behavior
- `docs/releases/v0.3.0.md` release notes
- examples that show a normal watchlist and a valuation-aware watchlist

## Release Criteria

v0.3.0 is ready when:

1. Existing `v0.2.0` commands still work.
2. `workflow_summary.json` contains step-level status and data reliability summary.
3. Single-stock reports explain data source status and valuation assumptions.
4. Dashboard exposes workflow status, warnings, failures, and retry hints.
5. Tests cover the reliability model, workflow summary, report, dashboard, and CLI paths.
6. README and release notes explain the new reliability model clearly.
