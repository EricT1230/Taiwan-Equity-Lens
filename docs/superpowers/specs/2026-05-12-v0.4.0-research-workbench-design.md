# v0.4.0 Research Workbench Design

## Goal

v0.4.0 turns Taiwan Equity Lens from a report generator into a local research workbench for tracking a group of Taiwan equities through repeatable research states.

## Product Positioning

The project remains a local CLI and static-report workflow. It should help a user organize research work, identify incomplete data or assumptions, and revisit watchlist names systematically. It must not become an investment-advice, portfolio-allocation, or trading-decision engine.

For a user running v0.4.0, the tool should answer:

1. Which stocks are in the current research universe?
2. What research state is each stock in?
3. Which stocks need manual valuation assumptions?
4. Which stocks have data reliability warnings or errors?
5. Which outputs should the user review next?

## Scope

Included:

- a research CSV model for tracking stock id, company name, category, priority, research state, and notes
- a `research` CLI command group with template creation, workflow execution, and summary generation
- a research summary JSON output that combines research rows with workflow reliability state
- dashboard rendering for research status, incomplete assumptions, and warning/error counts
- examples and documentation for a reproducible local research workflow

Excluded:

- hosted web service
- database or user accounts
- investment advice, buy/sell/hold labels, or allocation guidance
- automatic order execution
- external news, technical analysis, or chip-flow data
- long-running scheduler or daemon

## User Experience

### Initialize a Research File

The user can create a starter research file:

```powershell
python -m taiwan_stock_analysis.cli research init --output research.csv
```

The generated CSV should be immediately editable in a spreadsheet.

Required columns:

- `stock_id`
- `company_name`
- `category`
- `priority`
- `research_state`
- `notes`

Allowed `research_state` values:

- `new`
- `watching`
- `review`
- `done`
- `blocked`

Allowed `priority` values:

- `high`
- `medium`
- `low`

### Run Research Workflow

The user can run the existing watchlist workflow from the research file:

```powershell
python -m taiwan_stock_analysis.cli research run research.csv --output-dir research-dist --offline-prices
```

The command should:

- convert the research CSV into a watchlist input
- run the existing workflow pipeline
- preserve the research metadata
- write a research summary JSON
- write or refresh a dashboard that links to reports and workflow outputs

### Summarize Existing Outputs

The user can regenerate the research summary without rerunning financial data fetches:

```powershell
python -m taiwan_stock_analysis.cli research summary research.csv --workflow-dir research-dist --output research_summary.json
```

This path supports manual review after a workflow has already been generated.

## Architecture

Keep the current no-database architecture. Add a small research layer that adapts existing workflow outputs instead of duplicating analysis logic.

Recommended units:

- `src/taiwan_stock_analysis/research.py`
  - validates research CSV rows
  - writes the starter template
  - converts research rows to temporary watchlist rows
  - builds research summary dictionaries
- `src/taiwan_stock_analysis/cli.py`
  - adds the `research` command group with `init`, `run`, and `summary`
- `src/taiwan_stock_analysis/dashboard.py`
  - discovers and renders research summary outputs
- `tests/test_research.py`
  - covers CSV validation, template creation, and summary generation
- `tests/test_cli.py`
  - covers CLI routing for the `research` command group
- `tests/test_dashboard.py`
  - covers research summary dashboard rendering

No new third-party dependency should be introduced.

## Data Model

### Research CSV Row

```csv
stock_id,company_name,category,priority,research_state,notes
2330,TSMC,Semiconductor,high,review,Needs manual dividend assumption
2303,UMC,Semiconductor,medium,watching,Track valuation gap
```

Validation rules:

- `stock_id` is required.
- `company_name` may be blank.
- `category` may be blank.
- `priority` defaults to `medium` when blank.
- `research_state` defaults to `new` when blank.
- invalid `priority` raises a clear `ValueError`
- invalid `research_state` raises a clear `ValueError`

### Research Summary JSON

```json
{
  "research_path": "research.csv",
  "workflow_summary_path": "research-dist/workflow_summary.json",
  "counts": {
    "total": 2,
    "by_state": {
      "review": 1,
      "watching": 1
    },
    "by_priority": {
      "high": 1,
      "medium": 1
    },
    "needs_attention": 1
  },
  "items": [
    {
      "stock_id": "2330",
      "company_name": "TSMC",
      "category": "Semiconductor",
      "priority": "high",
      "research_state": "review",
      "notes": "Needs manual dividend assumption",
      "workflow_status": "ok",
      "reliability_status": "warning",
      "attention_reasons": ["valuation input has warning"]
    }
  ]
}
```

The summary should preserve unknown future workflow fields by reading them only for status extraction and keeping research metadata as the primary source of truth.

## Attention Rules

An item needs attention when any of these conditions apply:

- `research_state` is `new`, `review`, or `blocked`
- workflow status for the stock is `error`
- data reliability status for the stock or workflow is `warning` or `error`
- valuation output is skipped or unavailable

The wording must stay research-oriented. Use labels such as `needs review`, `missing assumption`, or `data warning`; do not use buy/sell/hold language.

## Error Handling

- Missing required `stock_id` column should raise `ValueError("research CSV must include a stock_id column")`.
- Invalid state should name the allowed states.
- Invalid priority should name the allowed priorities.
- Missing workflow summary during `research summary` should still produce a research summary with workflow status set to `skipped`.
- A failed stock in the workflow should not stop summary generation.

## Dashboard

The dashboard should add a research section when `research_summary.json` is found.

It should show:

- total research items
- counts by research state
- counts by priority
- items needing attention
- links to workflow summary, reports, and generated dashboard when available

The dashboard must remain static HTML and reuse the current visual style.

## Release Criteria

v0.4.0 is ready when:

1. Users can create a research CSV template from the CLI.
2. Users can run a research workflow from that CSV.
3. Users can generate a research summary without rerunning source fetches.
4. Dashboard renders research state and attention items.
5. Existing workflow, batch, valuation, comparison, and dashboard commands remain compatible.
6. Documentation explains the full research workflow from GitHub.
7. All tests pass.
