# v0.14.0 Review Action Layer Design

## Goal

v0.14.0 turns existing research status, source-audit findings, reliability warnings, and valuation gaps into deterministic review actions. A user should be able to open `research_summary.json`, the research pack, or the dashboard and know what must be checked next before handoff.

This release also updates the GitHub Actions workflow runtime actions to Node 24-compatible major versions so CI no longer emits Node.js 20 deprecation warnings.

## Product Positioning

Taiwan Equity Lens already shows source freshness, manual-review status, workflow failures, and research metadata. The next step is not more raw signals; it is a clear action layer that converts those signals into a review checklist.

The action layer is a workflow control surface. It does not produce buy, sell, hold, target allocation, ranking, or final investment decisions. It answers operational research questions:

- Which stocks need source verification?
- Which stocks have failed or skipped workflow stages?
- Which stocks need valuation assumptions completed?
- Which high-priority research items still need thesis, risk, trigger, or follow-up fields?
- Which actions should be reviewed first?

## Scope

Included:

- A focused review-action builder module.
- Per-stock `review_actions` in `research_summary.json`.
- Aggregate action counts and priority queues in `research_summary.json`.
- Pack rendering for a `Review Action Checklist`.
- Dashboard rendering for action counts and per-stock open actions.
- Documentation, changelog, release note, and version update to `v0.14.0`.
- GitHub Actions runtime update from Node 20-backed actions to Node 24-backed action majors.

Excluded:

- New external data vendors.
- Automatic data refetching.
- Background scheduling.
- Interactive dashboard filtering.
- State persistence for marking actions done.
- Investment recommendation language.

Interactive filtering and persisted action status are good future candidates, but v0.14.0 should first establish a stable deterministic action data model.

## Review Action Data Model

Each research summary item gets a `review_actions` list:

```json
{
  "stock_id": "2330",
  "review_actions": [
    {
      "id": "source-audit-manual-review",
      "category": "source_audit",
      "severity": "manual_review",
      "message": "Review source audit: fixture source requires manual review; offline mode",
      "status": "open"
    }
  ]
}
```

Fields:

- `id`: deterministic slug for the action type.
- `category`: one of `source_audit`, `workflow`, `reliability`, `valuation`, or `research_quality`.
- `severity`: one of `error`, `stale`, `unknown`, `manual_review`, `warning`, or `info`.
- `message`: short human-readable instruction.
- `status`: always `open` in v0.14.0.

The model intentionally avoids due dates, owners, comments, or persisted completion state. Those require a broader workflow storage design and should not be added in this release.

## Action Rules

Source-audit actions:

- `source_audit_status = stale` creates a `stale` action requiring data refresh or source verification.
- `source_audit_status = unknown` creates an `unknown` action requiring source metadata review.
- `source_audit_status = manual_review` creates a `manual_review` action using `source_audit_reasons`.

Workflow actions:

- `workflow_status = error` creates an `error` action with workflow failure context where available.
- `workflow_status = warning` creates a `warning` action to inspect partial or fallback workflow output.
- `workflow_status = skipped` creates an `info` action when the workflow did not run or lacked prerequisites.
- If workflow failure details are available, the action message should include the failed stage and reason.

Reliability actions:

- `reliability_status = error` creates an `error` action.
- `reliability_status = warning` creates a `warning` action.

Valuation actions:

- If valuation output is unavailable or skipped, create a `valuation` action.
- If existing attention reasons already include valuation output unavailable/skipped, convert that into a structured action instead of relying only on text.

Research-quality actions:

- High-priority items missing `thesis` create a `research_quality` action.
- High-priority items missing `follow_up_questions` create a `research_quality` action.
- Items in `new`, `review`, or `blocked` states create actions appropriate to that state.

Rules must be deterministic and should not duplicate equivalent actions for the same stock.

## Aggregates

`research_summary.json` adds top-level action metadata:

```json
{
  "review_action_summary": {
    "total_open": 4,
    "by_category": {
      "source_audit": 2,
      "valuation": 1,
      "research_quality": 1
    },
    "by_severity": {
      "manual_review": 2,
      "warning": 1,
      "info": 1
    }
  },
  "review_action_queue": [
    {
      "stock_id": "2330",
      "company_name": "TSMC",
      "priority": "high",
      "actions": [
        {
          "id": "source-audit-manual-review",
          "category": "source_audit",
          "severity": "manual_review",
          "message": "Review source audit: fixture source requires manual review; offline mode",
          "status": "open"
        }
      ]
    }
  ]
}
```

Queue ordering:

1. Highest severity first: `error`, `stale`, `unknown`, `manual_review`, `warning`, `info`.
2. Research priority: `high`, `medium`, `low`.
3. `stock_id` ascending.

## Pack Design

Research packs add a `Review Action Checklist` section after `Source Audit Overview` and before the priority review queue.

Markdown output:

```markdown
## Review Action Checklist
- Total open: 4
- By severity: error: 1, manual_review: 2, warning: 1

| Stock | Priority | Severity | Category | Action |
| --- | --- | --- | --- | --- |
| 2330 | high | manual_review | source_audit | Review source audit: fixture source requires manual review |
```

HTML output adds the same information in escaped table form.

## Dashboard Design

Dashboard rendering adds a `Review Actions` section when a research summary includes `review_action_summary` or per-item `review_actions`.

It should show:

- total open action count
- counts by severity and category
- per-stock action rows with stock id, priority, severity, category, and message

Missing action metadata in older summaries must render as absent, not broken.

## GitHub Actions Runtime Update

The current workflow uses:

- `actions/checkout@v4`
- `actions/setup-python@v5`

GitHub Actions now emits Node.js 20 deprecation warnings for those versions. v0.14.0 updates the workflow to Node 24-compatible action majors:

- `actions/checkout@v6`
- `actions/setup-python@v6`

The Python matrix remains `3.10`, `3.11`, and `3.12`. The release verification path remains `python -m unittest discover -s tests -v`.

## Error Handling

- Missing workflow payloads should keep `workflow_status = skipped` and create a single `workflow` / `info` action that tells the user to run or attach workflow outputs.
- Missing source-audit data should not create source-audit actions unless an item explicitly has unknown/manual-review/stale status.
- Malformed action inputs should be normalized or ignored rather than stringifying lists or dicts into user-facing messages.
- Pack and dashboard renderers must escape HTML and normalize Markdown table cells.

## Testing

Required coverage:

- Review-action builder creates deterministic actions from source audit, workflow, reliability, valuation, and research-quality conditions.
- Research summary items include `review_actions`.
- Research summary includes aggregate counts and ordered `review_action_queue`.
- Pack Markdown and HTML render `Review Action Checklist`.
- Dashboard renders review action counts and rows.
- Legacy summaries without review actions remain compatible.
- GitHub Actions workflow references Node 24-compatible action majors.
- Release doctor passes for `0.14.0`.

## Release Criteria

v0.14.0 is ready when:

1. Research summaries expose deterministic per-stock review actions.
2. Packs and dashboards surface the same action checklist without adding investment recommendation language.
3. Existing v0.13.0 source-audit and reliability workflows remain backward-compatible.
4. CI workflow no longer uses Node.js 20-backed `checkout` or `setup-python` action majors.
5. Local tests and GitHub Actions pass for `main` and the `v0.14.0` tag.
