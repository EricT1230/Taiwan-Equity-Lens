# v0.36.0 Expert Console State Sync Design

## Goal

v0.36.0 closes the v0.35.0 guided-flow loop. After a user handles a review action in a served dashboard, the Expert Agent Console should update immediately instead of telling the user to refresh.

## Scope

- Served dashboard mode:
  - A row-level or bulk `標記完成` / `稍後處理` update changes the row state through `/api/review-actions/set`.
  - After the row state changes, the Expert Agent Console recalculates from the existing review-action table rows.
  - The handoff verdict, open count, and Top 3 blocker list update in-place.
  - When no open actions remain, the console switches to manual handoff readiness.
- Static dashboard mode:
  - The console keeps the refresh/regenerate notice.
  - Static buttons still copy CLI commands only.

## Non-Goals

- Do not add a second state store.
- Do not add complete/defer/ignore controls directly inside the console.
- Do not imply investment advice, buy/sell/hold guidance, or position advice.
- Do not change existing review-action API payloads unless strictly required.

## Implementation Notes

- The review-action table remains the single source of truth in the browser.
- Add row metadata needed for console rebuilds:
  - `data-company-name`
  - `data-priority-label`
  - `data-severity-label`
  - `data-action-message`
  - existing `data-stock-id`, `data-category`, `data-category-label`, and `data-action-id`
- Add client-side helpers:
  - `syncExpertConsole(section)`
  - `renderExpertConsoleActions(consoleBlock, openRows)`
  - `attachExpertConsoleFocus(button)`
- Preserve all existing `data-review-*` hooks.

## Acceptance Tests

- Served dashboards include a sync note and the `syncExpertConsole(section)` call after API updates.
- Static dashboards keep the refresh/regenerate notice.
- Review-action rows expose enough escaped metadata to rebuild console actions.
- The console rebuild path targets actions by `stock_id + action_id`, not just by stock/category.
- Existing dashboard and dashboard-server tests continue to pass.
