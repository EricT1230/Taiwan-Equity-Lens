# v0.8.0 Traceability Layer Design

## Goal

v0.8.0 adds lightweight provenance metadata across workflow, research summary, memo summary, pack summary, and dashboard outputs so every generated artifact can be traced back to its inputs and related outputs.

## Product Positioning

Taiwan Equity Lens should remain a local file-based workflow. The traceability layer is not a database or audit platform. It is structured metadata that helps a user inspect:

1. Which run produced this output?
2. Which inputs were used?
3. Which downstream artifacts were derived from it?
4. Whether the current memo / pack belongs to the same research run?

## Scope

Included:

- run metadata with `run_id`, `generated_at`, and command context
- artifact registry fields in:
  - `workflow_summary.json`
  - `research_summary.json`
  - `memo_summary.json`
  - `pack_summary.json`
- dependency links between workflow, research, memo, and pack summaries
- dashboard rendering for traceability metadata when present
- documentation, release notes, version bump, and tests

Excluded:

- database persistence
- external lineage service
- cryptographic signing
- full reproducibility snapshotting of network payloads
- migration tooling for older summary files

## Metadata Contract

### Shared Run Metadata

Use a compact metadata object:

```json
{
  "run_id": "workflow-20260513T120000Z",
  "generated_at": "2026-05-13T12:00:00Z",
  "kind": "research_run",
  "command": "research run",
  "inputs": {
    "research_csv": "research.csv",
    "watchlist_csv": "research-dist/research_watchlist.csv"
  },
  "output_root": "research-dist"
}
```

### Artifact Registry

Each summary should expose:

```json
{
  "run_metadata": {},
  "artifact_registry": {
    "self": "research-dist/research_summary.json",
    "dependencies": {
      "workflow_summary": "research-dist/workflow_summary.json"
    },
    "outputs": {
      "memo_summary": "research-dist/memos/memo_summary.json",
      "pack_summary": "research-dist/packs/pack_summary.json"
    }
  }
}
```

The exact keys vary by artifact type, but the shape should stay consistent:

- `self`
- `dependencies`
- `outputs`

## Artifact-Specific Expectations

### Workflow Summary

Add:

- `run_metadata`
- `artifact_registry.self`
- `artifact_registry.outputs.dashboard`
- `artifact_registry.outputs.research_summary` when relevant consumers add it later

### Research Summary

Add:

- `run_metadata`
- `artifact_registry.self`
- `artifact_registry.dependencies.workflow_summary`
- `artifact_registry.outputs.memo_summary`
- `artifact_registry.outputs.pack_summary`

### Memo Summary

Add:

- `run_metadata`
- `artifact_registry.self`
- `artifact_registry.dependencies.research_summary`
- `artifact_registry.dependencies.workflow_summary`
- `artifact_registry.outputs.memo_markdown[]`
- `artifact_registry.outputs.memo_html[]`

### Pack Summary

Add:

- `run_metadata`
- `artifact_registry.self`
- `artifact_registry.dependencies.research_summary`
- `artifact_registry.dependencies.workflow_summary`
- `artifact_registry.dependencies.memo_summary`
- `artifact_registry.outputs.markdown`
- `artifact_registry.outputs.html`

## Run ID Strategy

Use a deterministic-enough local string built from UTC timestamp:

```text
<kind>-YYYYMMDDTHHMMSSZ
```

No UUID dependency is required.

If a downstream artifact is produced from an existing summary file that already has `run_metadata.run_id`, it should reuse that run id rather than invent a new one.

## Architecture

Add a small traceability helper module:

- `src/taiwan_stock_analysis/traceability.py`
  - `build_run_metadata(...)`
  - `read_run_metadata(payload)`
  - `build_artifact_registry(...)`
  - `with_traceability(payload, ...)`

Wire through:

- `workflow.py`
- `research.py`
- `memo.py`
- `pack.py`
- `dashboard.py`

No new third-party dependencies.

## Dashboard Behavior

When traceability fields exist, dashboard sections should render:

- run id
- generated timestamp
- artifact dependency links where relevant

At minimum:

- workflow summary row shows run id
- research summary section shows run id and workflow dependency
- memo output section can show summary link with inherited run id when available
- pack output section can show summary link with inherited run id when available

Older outputs without traceability fields should still render without failure.

## Error Handling

- Missing traceability fields must not break old files.
- Invalid traceability structures should degrade to empty strings / blank values in rendering.
- Downstream generators should reuse valid upstream run metadata when possible, but generate fresh metadata if none exists.

## Testing

Required coverage:

- workflow summary includes run metadata and artifact registry
- research summary inherits workflow run id and links back to workflow summary
- memo summary inherits run id and records research/workflow dependencies
- pack summary inherits run id and records dependencies and outputs
- dashboard renders run id / traceability fields when present
- dashboard remains compatible with older summaries without traceability fields
- existing tests continue passing

v0.8.0 is ready when:

1. Summary files expose traceability metadata consistently.
2. Downstream summaries reuse upstream run ids.
3. Dashboard surfaces traceability information without breaking old artifacts.
4. Tests cover metadata generation, inheritance, and rendering.
5. Documentation explains the traceability layer clearly.
