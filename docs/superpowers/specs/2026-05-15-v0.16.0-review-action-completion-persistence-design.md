# v0.16.0 Review Action Completion Persistence Design

## Goal

Persist manual review-action decisions across regenerated research outputs so the dashboard can be used as a working review queue, not only a static list of open items.

The first version stores completion state in a local sidecar JSON file. This keeps generated research summaries deterministic while preserving human decisions across reruns.

## Non-Goals

- Do not write manual completion state back into `research_summary.json`.
- Do not require a web server or backend service.
- Do not add browser write-back from the static dashboard.
- Do not remove open actions from generated summaries; instead, overlay persisted state at display and CLI list time.

## Recommended Approach

Add a sidecar state file named `review_action_state.json` in the research or workflow output directory.

Example:

```json
{
  "version": 1,
  "actions": {
    "2330:source-audit-manual-review": {
      "stock_id": "2330",
      "action_id": "source-audit-manual-review",
      "status": "done",
      "note": "checked source freshness",
      "updated_at": "2026-05-15T09:00:00Z"
    }
  }
}
```

The stable key is `<stock_id>:<action_id>`. This matches the deterministic action IDs introduced in v0.14.0 and remains independent of queue ordering.

Allowed persisted statuses:

- `open`
- `done`
- `deferred`
- `ignored`

`open` means a persisted override exists but returns the action to the open queue. Missing state is also treated as `open`.

## Architecture

Create a focused module:

- `src/taiwan_stock_analysis/review_action_state.py`

Responsibilities:

- Load and validate a sidecar state file.
- Write a normalized sidecar state file.
- Build stable action keys.
- Apply persisted state to review-action queues.
- Summarize persisted counts by status.

Keep `review_actions.py` responsible for generated action detection only. Keep `research.py` responsible for generated summaries only. This avoids coupling manual state to repeatable research generation.

## CLI

Add `research action` subcommands:

```bash
python -m taiwan_stock_analysis.cli research action list research-dist/research_summary.json --state research-dist/review_action_state.json
python -m taiwan_stock_analysis.cli research action set research-dist/review_action_state.json 2330 source-audit-manual-review --status done --note "checked"
```

`list` should read a generated `research_summary.json`, overlay state when provided, and print deterministic text rows with stock ID, priority, status, severity, category, action ID, and message.

`set` should create or update the sidecar file. It should validate the status and write stable JSON with sorted keys. It does not need the research summary because users may mark known action IDs from the dashboard or list output.

## Dashboard

When `write_dashboard_index()` discovers a `research_summary.json`, it should look for a sibling `review_action_state.json`. If found, the dashboard should overlay status onto the review-action rows.

Dashboard changes:

- Add a `status` column to the Review Actions table.
- Add `data-status` to each action row.
- Include state counts such as `state: done: 1, deferred: 1`.
- Extend the existing local filter bar with a status filter.

Actions with persisted `done`, `deferred`, or `ignored` should remain visible by default in v0.16.0. This preserves auditability and avoids surprising disappearance. Users can filter status to focus on open items.

## Error Handling

- Missing state file is valid and means all actions are open.
- Invalid JSON should not break dashboard generation. The dashboard should show generated actions as open and include a visible warning near the Review Actions section.
- Malformed state entries should be ignored.
- Unknown statuses should be ignored.

CLI `set` should fail with a clear error for invalid statuses.

## Testing

- Applying state to queue actions.

- Dashboard status column, status filter, state counts, invalid-state warning, and escaped attributes.
