# v0.25.0 Demo Doctor Design

## Goal

Add a first-run quality gate for the bundled demo. v0.24.0 made the demo easy to run with `demo quickstart`; v0.25.0 should make it equally easy to verify that the demo completed and produced the expected handoff surfaces.

## User Story

After running:

```powershell
python -m taiwan_stock_analysis.cli demo quickstart
```

a user can run:

```powershell
python -m taiwan_stock_analysis.cli doctor demo --output-dir demo-dist
```

and get a clear pass/fail summary for the generated demo output.

## Scope

Add a new `doctor demo` subcommand that checks the output directory generated by `demo quickstart`.

The command should validate required files, parse key JSON outputs, inspect expected summary content, and give a concrete repair hint when the demo is incomplete.

## Required Output Files

`doctor demo --output-dir demo-dist` should check:

- `dashboard.html`
- `workflow_summary.json`
- `research_summary.json`
- `memos/memo_summary.json`
- `packs/pack_summary.json`
- `comparison/comparison.html`
- `comparison/comparison.json`
- `valuation.csv`

## Content Checks

The doctor should report failure when:

- a required file is missing,
- `workflow_summary.json` is invalid JSON,
- `research_summary.json` is invalid JSON,
- `research_summary.json` has no review-action queue,
- `dashboard.html` does not include `Review Actions`,
- `workflow_summary.json` has no successful stock ids.

The doctor should not perform live network calls or mutate the output directory.

## CLI Output

Successful run:

```text
Demo readiness OK:
- output directory demo-dist
- required files present
- workflow summary has successful stocks
- research summary has review actions
- dashboard includes Review Actions
```

Failed run:

```text
Demo readiness failed:
- missing demo-dist/dashboard.html
- invalid JSON: demo-dist/research_summary.json

Repair:
python -m taiwan_stock_analysis.cli demo quickstart --output-dir demo-dist
```

The exact text can follow local CLI style, but it must include:

- an OK or failed headline,
- concrete failed checks,
- the repair command.

## Architecture

Prefer adding a small doctor helper in `taiwan_stock_analysis.doctor` rather than keeping validation logic inside `cli.py`.

Suggested shape:

- `DemoDoctorResult` dataclass with `ok`, `messages`, and `repair_command`.
- `check_demo_readiness(output_dir: Path) -> DemoDoctorResult`.
- `format_demo_doctor_result(result: DemoDoctorResult) -> str`.
- `cli.py` wires `doctor demo --output-dir` and returns `0` on pass, `1` on fail.

This mirrors the existing release doctor pattern while keeping the CLI layer thin.

## Documentation Updates

Update:

- `README.md`: show `demo quickstart`, then `doctor demo`.
- `docs/usage-workflow.md`: add the demo doctor check after the fastest local demo command.
- `examples/README.md`: mention the doctor command after opening the dashboard.
- `CHANGELOG.md` and `docs/releases/v0.25.0.md` during release metadata.

## Non-Goals

- No new dashboard UI controls.
- No automatic browser opening.
- No live Goodinfo or market-price checks.
- No auto-repair mode.
- No deletion or rewriting of demo outputs.
- No investment recommendation or scoring change.

## Acceptance Criteria

- `python -m taiwan_stock_analysis.cli demo quickstart --output-dir .tmp-cli-test/v0.25-demo-doctor` succeeds.
- `python -m taiwan_stock_analysis.cli doctor demo --output-dir .tmp-cli-test/v0.25-demo-doctor` returns `0`.
- Missing required output returns `1` with a missing-file message.
- Invalid JSON returns `1` with an invalid-JSON message.
- Missing review-action queue returns `1`.
- Missing successful stock ids returns `1`.
- Dashboard without `Review Actions` returns `1`.
- Full unit suite passes.
- Release doctor passes for `0.25.0`.
