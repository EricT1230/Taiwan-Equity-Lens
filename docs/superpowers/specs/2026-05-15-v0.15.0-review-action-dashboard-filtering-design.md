# v0.15.0 Review Action Dashboard Filtering Design

## Goal

v0.15.0 makes the dashboard `Review Actions` section easier to use by adding client-side filters for action severity, category, priority, and free-text search.

This release keeps the v0.14.0 review-action data model unchanged. Filtering is a dashboard viewing aid only; it does not mutate `research_summary.json`, persist completion state, or change the meaning of `review_actions`.

## Product Positioning

v0.14.0 established deterministic review actions across research summaries, packs, and dashboards. The next useful step is to make the dashboard review queue scannable when a workflow has many stocks or many open actions.

The dashboard should answer operational questions quickly:

- Which actions are errors or stale data checks?
- Which actions are source-audit tasks versus valuation or research-quality tasks?
- Which high-priority items still need attention?
- Which rows mention a specific stock, category, or action phrase?

The feature remains research workflow support only. It must not introduce investment recommendation language, rankings, or action completion semantics.

## Scope

Included:

- Static dashboard filter controls for `Review Actions`.
- Severity filter with `all`, `error`, `stale`, `unknown`, `manual_review`, `warning`, and `info`.
- Category filter with `all`, `source_audit`, `workflow`, `reliability`, `valuation`, and `research_quality`.
- Priority filter with `all`, `high`, `medium`, and `low`.
- Text search across stock ID, priority, severity, category, and action message.
- Stable row metadata through escaped `data-*` attributes.
- Empty-state feedback when filters hide every action.
- Tests for filter controls, row metadata, escaping, and legacy compatibility.

Excluded:

- Persisted done/open state.
- Owners, due dates, comments, or action notes.
- CSV or JSON write-back.
- Server-side filtering.
- New frontend dependencies.
- Interactive sorting beyond the existing queue order.

Persisted action completion remains a future feature. This release should leave the HTML row metadata clean enough for a later state layer to build on it without changing the action data contract.

## UI Design

The dashboard adds a compact filter bar at the top of each `Review Actions` section:

- `Severity` select.
- `Category` select.
- `Priority` select.
- `Search` input.
- `Reset` button.
- A small count label showing visible rows versus total rows.

The controls should be ordinary form controls inside the existing dashboard style. They should not require a build step or external assets.

When there are no matching rows, the table body should show one empty-state row:

```text
No review actions match the current filters.
```

If a dashboard has no review actions at all, the entire `Review Actions` section remains absent, matching v0.14.0 legacy behavior.

## Data Model

No JSON data model changes are required.

Each rendered action table row adds normalized metadata:

```html
<tr
  data-review-action-row="true"
  data-stock-id="2330"
  data-priority="high"
  data-severity="manual_review"
  data-category="source_audit"
  data-search-text="2330 high manual_review source_audit review source audit fixture source"
>
```

All attribute values must be HTML-escaped. Text search should be case-insensitive and should use the normalized `data-search-text` value.

The table retains the current visible columns:

- `stock_id`
- `priority`
- `severity`
- `category`
- `action`

## Behavior

On dashboard load:

1. Find each review-action section.
2. Attach change/input handlers to the filter controls.
3. Filter rows in place by comparing each row's metadata with selected values.
4. Update visible count.
5. Show or hide the filter empty-state row.

Filtering rules:

- `all` means no filtering for that dimension.
- Severity, category, and priority filters require exact matches.
- Search matches if the lowercase query is contained in `data-search-text`.
- Multiple filters combine with AND semantics.
- Reset returns all selects to `all`, clears search, and shows all rows.

The script should tolerate missing controls or malformed rows by skipping that section rather than throwing.

## Architecture

`dashboard.py` owns the feature because the dashboard already renders static HTML and embeds small JavaScript for command builders.

Implementation shape:

- Extend `_review_actions_section()` to emit a filter bar and a section-level wrapper class or data attribute.
- Extend `_review_action_rows()` to include escaped row metadata.
- Add a small helper to normalize text for row metadata.
- Add an embedded JavaScript function near the existing dashboard script.
- Keep filtering code independent from command builder code.

No changes are needed in `research.py`, `review_actions.py`, `pack.py`, or workflow summary generation.

## Error Handling

- Missing `review_action_summary` should not break rendering if `review_action_queue` exists.
- Missing or malformed `actions` arrays should continue to render an empty table body row.
- Missing filter controls in the DOM should be ignored by JavaScript.
- User-provided values in stock IDs, categories, and messages must remain escaped in visible HTML and attributes.

## Testing

Required coverage:

- Dashboard renders filter controls when review actions exist.
- Dashboard does not render filter controls for legacy summaries without review actions.
- Review action rows include stable escaped `data-*` attributes.
- Search text metadata includes stock ID, priority, severity, category, and message content.
- Review action row escaping still protects visible cells and attributes.
- Embedded JavaScript includes the filtering hooks, reset behavior, and no-match empty state.
- Existing dashboard tests continue to pass.

## Release Criteria

v0.15.0 is ready when:

1. The dashboard can filter review actions by severity, category, priority, and search text.
2. Filtering works in a static HTML file with no dependencies.
3. Legacy dashboards without review-action data remain compatible.
4. The review-action JSON data model remains unchanged.
5. Local tests and GitHub Actions pass for `main` and the `v0.15.0` tag.

## Future Persistence Path

A later release can add persisted completion state without changing this release's filter contract by reading a separate local state artifact, such as `review_actions_state.json`, and joining it to rendered rows by stock ID plus action ID.

That later state model should decide whether completion belongs in a sidecar JSON file, a user-maintained CSV column, or a dedicated local workflow store. v0.15.0 intentionally avoids that decision.
