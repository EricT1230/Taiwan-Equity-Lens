# v0.18.0 Review Action State Report Design

## Goal

Add a read-only audit layer for persisted review-action state so users can understand queue health, hand off review status, and spot stale sidecar entries.

v0.16.0 added `review_action_state.json`. v0.17.0 made dashboard state updates easier by copying CLI commands. v0.18.0 makes the state inspectable and auditable without mutating it.

## Non-Goals

- Do not delete stale state entries.
- Do not add `research action prune`.
- Do not change the `review_action_state.json` schema.
- Do not make the dashboard write state files.
- Do not turn this into portfolio advice or action prioritization beyond the existing queue order.

## CLI Report

Add:

```powershell
python -m taiwan_stock_analysis.cli research action report research-dist/research_summary.json --state research-dist/review_action_state.json
```

The report reads `research_summary.json`, overlays state from the sidecar file, and prints deterministic text output.

Required output fields:

- `total_actions`: count of current generated review actions.
- `by_status`: counts for `open`, `done`, `deferred`, and `ignored`.
- `stale_state`: count of state entries that are not present in the current generated queue.
- `last_updated`: max `updated_at` value from valid state entries, or `-`.
- `next_open`: first several current open actions in queue order.

The default state path is the sibling `review_action_state.json` next to `research_summary.json`, matching `research action list`.

## State Report Model

Add helpers to `review_action_state.py`:

- Build the set of current action keys from a queue.
- Build stale state rows by comparing state entries against current action keys.
- Build a summary object with status counts, stale count, last updated, and next open rows.

Suggested shape:

```python
{
    "total_actions": 7,
    "by_status": {"open": 6, "done": 1},
    "stale_state": [
        {"stock_id": "9999", "action_id": "workflow-error", "status": "done", "updated_at": "..."}
    ],
    "last_updated": "2026-05-15T09:00:00Z",
    "next_open": [
        {"stock_id": "2330", "action_id": "reliability-warning", "severity": "warning", ...}
    ],
}
```

`next_open` should follow the existing generated queue order and be capped for CLI and dashboard readability.

## Dashboard Health

In the dashboard Review Actions section, add a compact state health line near the existing badges:

- `state health: open: 6, done: 1`
- `stale state: 2`
- `last updated: 2026-05-15T09:00:00Z`

If the sidecar state is invalid, keep the current v0.16 warning behavior. The health line should treat invalid or missing state as all current generated actions being open and stale count zero.

Stale state entries should not be shown in the main action table because they no longer correspond to current generated actions. The dashboard should only summarize the count in v0.18.0.

## Error Handling

- Missing state file is valid and reports all current actions as open.
- Invalid state JSON should not break the CLI report or dashboard. The CLI report should print a warning and continue with empty state.
- Malformed state entries are ignored by the existing normalization path.
- Unknown statuses are ignored by the existing normalization path.
