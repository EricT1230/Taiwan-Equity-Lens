# v0.19.0 Prune Stale Review Action State Design

## Goal

Add an explicit, safe cleanup command for stale `review_action_state.json` entries. v0.18.0 can report stale sidecar entries; v0.19.0 lets the user remove those stale entries only when they intentionally request a write.

## User Interface

Add a `research action prune-stale` CLI command:

```powershell
python -m taiwan_stock_analysis.cli research action prune-stale research-dist/research_summary.json --state research-dist/review_action_state.json
python -m taiwan_stock_analysis.cli research action prune-stale research-dist/research_summary.json --state research-dist/review_action_state.json --write
```

When `--state` is omitted, the command uses the sibling `review_action_state.json` next to `research_summary.json`, matching `list` and `report`.

Default behavior is dry-run. It prints how many stale entries would be removed and lists them as:

```text
stock_id	status	action_id	updated_at	note
```

With `--write`, it rewrites the state file after removing only stale entries and prints:

```text
Pruned N stale review action state entries
```

## Data Flow

1. Read `research_summary.json`.
2. Extract `review_action_queue`; if the field is missing or malformed, treat it as an empty queue.
3. Load and normalize `review_action_state.json` with existing state-loading behavior.
4. Compute current review-action keys from the generated queue.
5. Identify normalized state entries whose keys are not in the current queue.
6. In dry-run mode, print stale entries without writing.
7. In `--write` mode, write a normalized state payload containing only non-stale entries.

## Safety And Error Handling

- The command is read-only unless `--write` is provided.
- It never removes entries whose `stock_id:action_id` still exists in the current generated queue.
- It keeps the existing state schema and `version`.
- If the state file cannot be parsed, the command prints the existing warning, does not write, and returns non-zero.
- Missing state file is allowed: dry-run reports zero stale entries; `--write` creates no new state file.
- Malformed state entries continue to be ignored by the existing normalization rules.

## Scope

In scope:

- State pruning helper in `review_action_state.py`.
- CLI command and deterministic text output.
- Tests for dry-run, write, default state path, no-stale behavior, and invalid state JSON.
- Release metadata and docs for v0.19.0.

Out of scope:

- Dashboard write actions.
- Automatic pruning during `research run` or dashboard generation.
- State schema changes.
- JSON or Markdown export for action reports.

## Testing

Unit coverage should verify:

- Dry-run lists stale entries and leaves the file unchanged.
- `--write` removes stale entries and preserves current entries.
- No stale entries produces a zero-count result and stable state.
- Invalid state JSON returns non-zero and does not modify the file.
- The default sibling state path works.

Release verification should include focused state/CLI tests, the full test suite, Python 3.10 smoke, release doctor, and a demo workflow that creates one current state entry plus one stale entry and prunes it with `--write`.
