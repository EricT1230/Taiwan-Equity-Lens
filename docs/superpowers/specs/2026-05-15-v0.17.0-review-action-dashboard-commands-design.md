# v0.17.0 Review Action Dashboard Commands Design

## Goal

Make persisted review-action state easier to use from the static dashboard by generating ready-to-run CLI commands for each review action.

v0.16.0 added durable `review_action_state.json` sidecar files and CLI commands. v0.17.0 keeps that persistence model and adds a dashboard command surface so users can move from seeing an action to copying the exact command needed to mark it done, deferred, ignored, or open again.

## Non-Goals

- Do not let the browser write `review_action_state.json` directly.
- Do not add a local web server.
- Do not add JavaScript dependencies or a frontend framework.
- Do not change the v0.16.0 sidecar schema.
- Do not add dashboard note editing in this release.

## Dashboard Behavior

The `Review Actions` table gains a `commands` column.

Each action row renders command buttons for:

- `done`
- `deferred`
- `ignored`
- `reopen`

`reopen` maps to the persisted status `open`.

Clicking a command button copies the corresponding command to the clipboard:

```powershell
python -m taiwan_stock_analysis.cli research action set review_action_state.json 2330 source-audit-manual-review --status done
```

The command should use the sibling state path discovered for the current research summary. If no state file exists yet, the command should still point to `review_action_state.json` next to `research_summary.json`.

Rows should also show persisted state metadata when available:

- `note`
- `updated_at`

The first version displays these values only. Editing notes remains a CLI task.

## Copy Interaction

The dashboard remains static HTML. JavaScript should only handle local filtering and copying text to the clipboard.

Copy behavior:

- Prefer `navigator.clipboard.writeText`.
- Fall back to a temporary `<textarea>` and `document.execCommand("copy")`.
- Update a small status label in the Review Actions section with the copied status, such as `Copied done command for 2330`.

The command should also remain present in the DOM, either as a `data-command` attribute or visible/fallback `<code>` text, so the workflow remains inspectable if clipboard access is blocked.

## Command Escaping

Commands must be safe for stock IDs, action IDs, paths, and notes that contain spaces or shell-sensitive characters.

Use a small PowerShell-oriented quoting helper for generated command arguments:

- Plain safe values can remain unquoted.
- Other values should be single-quoted.
- Embedded single quotes should be escaped for PowerShell by doubling them.

This release does not add `--note` to dashboard-generated commands because dashboard note editing is out of scope. Users can append `--note` manually or use the CLI directly.

## Architecture

Keep changes inside `dashboard.py` unless a small helper in `review_action_state.py` clearly removes duplication.

Recommended helpers:

- Resolve the state path for a research summary.
- Build review-action state commands.
- Quote command arguments for PowerShell.
- Render command buttons.
- Render state metadata.

Existing v0.16.0 state loading and overlay logic should stay unchanged.

## Testing

Add dashboard tests for:

- `commands` column appears in Review Actions.
- Each row includes commands for done, deferred, ignored, and reopen.
- `reopen` command uses `--status open`.
- Commands use the sibling `review_action_state.json` path.
- Persisted `note` and `updated_at` render when present.
- Clipboard JavaScript and fallback copy path are present.
- Command values are escaped in HTML attributes and visible fallback text.

Run focused dashboard tests, full unittest discovery, Python 3.10 `uv run`, and release doctor for `0.17.0`.
