# v0.10.0 Release Quality Gate Design

## Summary

v0.10.0 adds a local release quality gate so Taiwan Equity Lens can verify release readiness before a version commit or tag is pushed. The goal is to prevent mismatched version metadata, missing release notes, broken documentation links, and Python-version compatibility failures from escaping into GitHub Actions after a tag exists.

This release does not add new stock-analysis behavior. It improves the engineering workflow around releases.

## Goals

- Add a deterministic local quality command for release readiness checks.
- Verify that package version metadata, README badge, CHANGELOG, and release note files agree.
- Verify documentation links used by the public README and release files resolve inside the repo.
- Make GitHub Actions show every Python matrix result by disabling fail-fast.
- Add tests for the quality gate so future release metadata drift is caught by `unittest`.

## Non-Goals

- Do not publish packages to PyPI.
- Do not automate GitHub release creation.
- Do not rewrite the existing release history.
- Do not add external dependencies.
- Do not change financial calculations, memo generation, dashboard rendering, or research workflow behavior.

## User-Facing Behavior

The CLI will expose a new command:

```powershell
python -m taiwan_stock_analysis.cli doctor release
```

The command exits with code `0` when the repo is release-ready and prints a concise OK summary. It exits with code `1` when one or more checks fail and prints actionable failure lines.

Optional explicit version check:

```powershell
python -m taiwan_stock_analysis.cli doctor release --version 0.10.0
```

When `--version` is provided, the command verifies the requested version against:

- `pyproject.toml` project version
- README version badge
- `CHANGELOG.md` top release section
- `docs/releases/vX.Y.Z.md`
- README release-note link

Without `--version`, the command reads the version from `pyproject.toml` and applies the same checks.

## Checks

### Version Consistency

For version `0.10.0`, the command checks:

- `pyproject.toml` contains `version = "0.10.0"`.
- `README.md` contains `version-v0.10.0-blue.svg`.
- `CHANGELOG.md` contains a `## v0.10.0 - YYYY-MM-DD` section.
- `docs/releases/v0.10.0.md` exists.
- `README.md` links to `docs/releases/v0.10.0.md`.

### Documentation Link Integrity

The command scans Markdown links in:

- `README.md`
- `CHANGELOG.md`
- `docs/usage-workflow.md`
- `docs/releases/v0.10.0.md`

It validates local relative links and ignores external `http`, `https`, `mailto`, and anchor-only links. Missing local targets fail the check.

### Python Matrix Guidance

The project supports Python `>=3.10`, while CI runs 3.10, 3.11, and 3.12. The doctor command will report the current Python runtime version, but it will not try to install other Python versions. Multi-version runtime validation remains the job of CI and local `uv run --python 3.10 ...` checks.

## GitHub Actions Change

The test workflow matrix will set:

```yaml
fail-fast: false
```

This makes all Python versions finish independently. A single version failure will not cancel the remaining jobs, which improves diagnosis.

## Architecture

Add a focused `doctor.py` module for repo health checks. It will be pure standard library code, with no network calls and no dependency on generated output directories.

Responsibilities:

- parse `pyproject.toml` version via `tomllib`
- scan Markdown links with a small deterministic parser
- build structured check results
- render concise CLI output

`cli.py` will add a `doctor release` subcommand and delegate all release checking to `doctor.py`.

## Error Handling

The command should collect all failures before exiting. This is more useful than failing fast because release metadata issues often appear in groups.

Example failure output:

```text
Release readiness failed:
- README badge expected v0.10.0
- Missing release note: docs/releases/v0.10.0.md
- Broken link in README.md: docs/releases/v0.10.0.md
```

## Testing

Tests will cover:

- passing release checks using the current repository files
- mismatched README badge
- missing release note
- missing CHANGELOG section
- broken local Markdown link
- CLI exit code `0` on pass
- CLI exit code `1` on release check failure

The implementation should keep tests isolated by using temporary fixture directories instead of mutating real project files.

## Documentation

Update:

- README common commands with `doctor release`

## Acceptance Criteria

- `python -m taiwan_stock_analysis.cli doctor release --version 0.10.0` passes after v0.10.0 docs/version updates are complete.
- `python -m unittest discover -s tests -v` passes.
- `uv run --python 3.10 python -m unittest discover -s tests -v` passes.
- GitHub Actions matrix keeps 3.10, 3.11, and 3.12 visible even if one version fails.
