# Dashboard Polish Design

## Goal

Phase 16A improves the generated dashboard so it reads like a product surface instead of a raw engineering index. The scope is limited to the static dashboard HTML.

## Scope

The dashboard will use clear Traditional Chinese labels, stable empty states, and a more useful workflow overview. It will continue to be generated by `src/taiwan_stock_analysis/dashboard.py` with no frontend framework and no new runtime dependencies.

## User Experience

The first screen should answer four questions:

1. How many stock reports were generated?
2. How many batch rows succeeded or failed?
3. Did comparison run or was it skipped?
4. Where are the workflow summary and valuation CSV?

The page will keep existing report, comparison, batch, and workflow sections, but the section titles, table headers, and empty-state text will be normal Traditional Chinese. Command builders remain available but should no longer dominate the workflow status.

## Architecture

Keep discovery logic unchanged except where needed for rendering. Add small rendering helpers in `dashboard.py`:

- summary card rendering
- table empty-state rendering
- batch status labels
- workflow status labels

The existing `render_dashboard_html(items)` API remains unchanged.

## Non-Goals

- Do not change valuation calculations.
- Do not add charting libraries.
- Do not introduce React or another frontend framework.
- Do not add live browser behavior beyond the existing command text updates.
- Do not change `workflow_summary.json` schema.

## Testing

Tests should assert user-visible text and behavior:

- clean Traditional Chinese dashboard title and section labels
- empty dashboard states render without broken tables
- workflow summary shows total and successful stock counts
- batch rows show readable status labels
- existing links remain present
- full workflow dashboard contains workflow summary links
